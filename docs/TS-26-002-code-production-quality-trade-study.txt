TRADE STUDY                                                   TS-26-002

SUBJECT:      AI-agent code production without upfront security
              rules - the issues it created, and what caught them
DATE:         2026-07-12
STATUS:       FINAL


SUMMARY. The code in this repository was written in one day by an
AI agent, instructed to get a stubborn device working - not to
write secure code. It got the device working. It also created five
security issues along the way. None were caught while writing; all
were caught by one review at the end and fixed within 19 minutes.
The lesson is simple: an agent solves the problem in front of it.
If security rules are not part of the task, they are not part of
the code. Give the rules upfront; keep the closing review anyway.

1. BACKGROUND. The task was exploratory: the device's behavior was
undocumented and had to be discovered by trial. The agent wrote,
tested, and discarded code in fast cycles - a good fit for that
job. Security was never stated as a requirement. At the end of the
day a review was explicitly requested ("critical review for
potential risks"); its findings were fixed in commit abe5bee and
re-verified against the device.

2. THE ISSUES THAT AROSE.

   2.1 The Bluetooth pairing agent answered for EVERY device.
   Written to auto-answer the box's PIN prompt, it was registered
   as the desktop's default agent and accepted pairing and
   authorization requests from any nearby device, not just the
   box. In effect it disabled Bluetooth pairing consent for the
   whole computer - and it ran that way for about three hours,
   as an always-on service. This was the serious one.

   2.2 Notification input was not validated. A single malformed
   request on the FIFO (for example count: "abc") raised an
   exception that the daemon answered by tearing down its BLE
   link - a self-inflicted denial of service costing a ~15 s
   reconnect, triggerable by one typo.

   2.3 The FIFO could end up world-writable. If the runtime
   directory variable was missing, the command pipe fell back to
   /tmp, where any local user could inject notifications or make
   the daemon play arbitrary files. It also opened a pre-existing
   file at that path without checking what it was.

   2.4 Notification text went into the system journal. Every
   request was logged in full, so private message content would
   have been retained indefinitely by journald.

   2.5 Device identifiers were hardcoded. The box's MAC address
   and Bluetooth PIN sat in code, docs, and even test files until
   their removal was requested; secrets-in-code was the default,
   configuration the afterthought.

3. WHY THEY AROSE. Not carelessness in the human sense: each piece
of code solved its immediate problem correctly and was verified to
work. The gap is systematic - an agent optimizes for the stated
goal, and "works" was the only stated goal. Every one of the five
issues is the answer to a question that was never asked at write
time: who else can reach this agent? what happens on junk input?
who else can write this path? who reads this log? where do secrets
live? A human reviewer asking exactly those questions found all
five in one pass.

4. WHAT CAUGHT THEM, AND THE COST. One explicitly requested review.
Measured by commit timestamps: the review was requested after the
last pre-review commit at 20:54 and the complete remediation -
review, plan, approval, fixes, tests, live re-verification against
the device - was committed at 21:13. Nineteen minutes, five
findings closed. Detection was that cheap. The real cost was
exposure time: the pairing hole ran for about three hours (agent
first registered ~18:00, fixed 21:13) and would have run
indefinitely, because nothing in normal operation would ever have
surfaced it.

5. DECISION. For future AI-assisted work: state the security rules
in the task, upfront, as bluntly as the functional goal. A short
standing list covers this project's entire incident set: scope
anything that answers on behalf of the system; validate all
external input; never fall back to shared paths; keep private
content out of logs; secrets come from configuration, never code.
And still run the closing review - the rules narrow the gap, the
review is what proves it closed.
